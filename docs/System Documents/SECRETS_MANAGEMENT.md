**Version:** v11.0.1 · **Maintainer:** Quietfire AI

## What This Covers

TelsonBase uses Docker Secrets for all sensitive values - API keys, signing keys,
encryption keys, and all infrastructure passwords. This document explains how the
secrets system works, how to generate and rotate secrets, what each secret does,
and what happens if you skip any of it.

Related documents:
- `docs/System Documents/ENV_CONFIGURATION.md` - non-sensitive config values in `.env`
- `docs/Operation Documents/INSTALLATION_GUIDE_WINDOWS.md` - first-run setup walkthrough
- `docs/Backup and Recovery Documents/Restore_and_Recover_Guide.md` - backup procedures

## Secrets Inventory

Nine secrets are required. All are generated by `scripts/generate_secrets.sh`.

| Secret File | Purpose | Generated As |
|---|---|---|
| `telsonbase_mcp_api_key` | Master API key - authenticates all requests to the MCP server | 64 hex chars (openssl rand -hex 32) |
| `telsonbase_jwt_secret` | JWT token signing key (HS256) - signs every session token | 64 hex chars |
| `telsonbase_encryption_key` | AES-256-GCM master key - encrypts sensitive fields at rest | 64 hex chars |
| `telsonbase_encryption_salt` | PBKDF2 key derivation salt - used with encryption key | 32 hex chars |
| `telsonbase_webui_secret` | Open-WebUI session signing secret | 64 hex chars |
| `telsonbase_grafana_password` | Grafana admin dashboard password | 24-char alphanumeric |
| `telsonbase_postgres_password` | PostgreSQL database authentication | 32-char alphanumeric |
| `telsonbase_redis_password` | Redis server authentication | 32-char alphanumeric |
| `telsonbase_mqtt_password` | Mosquitto MQTT broker password | 24-char alphanumeric |

Your API key - the one you use to connect to the dashboard and authenticate API calls - is
`telsonbase_mcp_api_key`. After running the script:

```bash
cat secrets/telsonbase_mcp_api_key
```

## How Secrets Are Resolved in Code

`core/config.py` implements a three-layer resolution strategy via `_resolve_secret()`:

**Layer 1 - Docker secret file** (highest priority)
```
/run/secrets/<secret_name>
```
Docker Compose mounts each file from `./secrets/` into the container at this path. This is
the production path. Secrets resolved this way never appear in `docker inspect` output, never
enter the process environment, and are never written to container disk layers.

**Layer 2 - Environment variable** (`.env` or shell)
```
MCP_API_KEY, JWT_SECRET_KEY, POSTGRES_PASSWORD, ...
```
Falls through to this if the Docker secret file is not present. Used during development
and by the `.env` sync step in `generate_secrets.sh` (see below).

**Layer 3 - Hardcoded default** (lowest priority)
```
"CHANGE_ME_IN_PRODUCTION_GENERATE_WITH_OPENSSL"
```
Triggers a `warnings.warn()` in development. In production (`TELSONBASE_ENV=production`),
falling through to a hardcoded default raises `ValueError` and the application refuses to
start. See the [Production Mode](#production-mode-telsonbase_envproduction) section.

## Generating Secrets

### First Run (no `secrets/` directory yet)

Prerequisites: `.env` must exist before running the script. Copy it first:

```bash
cp .env.example .env
```

Then generate all secrets:

```bash
bash scripts/generate_secrets.sh
```

**On Windows:** Run this in **Git Bash** (not PowerShell, not Command Prompt). Git Bash
ships with the Git for Windows installer. Find it in your Start menu.

The script will:
1. Create `./secrets/` directory (mode 700)
2. Generate all 9 secret files (mode 644 - see permissions note below)
3. Sync generated values into your `.env` file
4. Generate the Mosquitto `password_file` via Docker

### If `secrets/` Already Exists

The script detects an existing directory and prompts:

```
WARNING: ./secrets/ already exists.

Options:
  1) Overwrite all secrets (DANGER: existing sessions/tokens will be invalidated)
  2) Generate only missing secrets (safe)
  3) Abort
```

Choose option 2 to add any missing secret files without touching existing ones.
Choose option 1 only if you intend a full rotation (equivalent to `--rotate`).

### Forced Rotation (`--rotate`)

```bash
bash scripts/generate_secrets.sh --rotate
```

Prompts for typed confirmation (`ROTATE`) before proceeding. Regenerates all 9 secrets.

**What rotation invalidates immediately:**
- All active JWT sessions - every logged-in user is logged out
- All API keys - any external integration using the old key will fail
- The encryption key - data encrypted with the old key cannot be decrypted with the new one
- All infrastructure passwords - stack must be fully restarted

**Rotation procedure:**
1. Back up the current `./secrets/` directory before rotating
2. Run `bash scripts/generate_secrets.sh --rotate` and type `ROTATE`
3. Restart the full stack: `docker compose down && docker compose up -d --build`
4. Verify all services are healthy: `docker compose ps`
5. Update any external integrations (Goose config, Claude Desktop config, monitoring) that
   reference the old API key

### Verification Only (`--check`)

```bash
bash scripts/generate_secrets.sh --check
```

Checks that all 9 secret files exist, are non-empty (>= 16 bytes), and have correct
permissions. No secrets are generated or modified. Use this to confirm a fresh deployment
or after a restore.

## File Permissions: Why 644, Not 600

Secret files are set to mode `644` (owner read/write, group and world read). This is
intentional. TelsonBase containers run as non-root users and need to read the secrets
mounted at `/run/secrets/`. Mode `600` would block that read and cause startup failures.

The `secrets/` **directory** is mode `700` (owner only). Files inside are readable but the
directory listing is restricted.

## The `.env` Sync Step

`generate_secrets.sh` syncs secret values into your `.env` file automatically. This ensures
Layer 2 resolution works correctly in development without manually copying values.

Variables synced:

| `.env` Variable | Source Secret File |
|---|---|
| `MCP_API_KEY` | `telsonbase_mcp_api_key` |
| `JWT_SECRET_KEY` | `telsonbase_jwt_secret` |
| `POSTGRES_PASSWORD` | `telsonbase_postgres_password` |
| `REDIS_PASSWORD` | `telsonbase_redis_password` |
| `MOSQUITTO_PASSWORD` | `telsonbase_mqtt_password` |
| `WEBUI_SECRET_KEY` | `telsonbase_webui_secret` |
| `DATABASE_URL` | Password portion updated inline |

The `.env` file is safe to commit **only if it contains no real secrets** - which is the
state of `.env.example`. After running `generate_secrets.sh`, your `.env` contains live
secret values and must not be committed. It is in `.gitignore`.

## Mosquitto Password File

The MQTT broker (Mosquitto) requires a hashed password file - it cannot read a plaintext
password directly. `generate_secrets.sh` generates this file automatically using Docker:

```
monitoring/mosquitto/password_file
```

This file is excluded from the tarball and from git. If you extract a new release archive
over an existing installation, the password file will not be overwritten - it stays in sync
with whatever password was last generated by the script.

If you need to regenerate it manually (e.g., after a server restore):

```bash
MQPASS=$(cat secrets/telsonbase_mqtt_password)
rm -f monitoring/mosquitto/password_file
docker run --rm \
  -v $(pwd)/monitoring/mosquitto:/mosquitto/config \
  eclipse-mosquitto:2 \
  mosquitto_passwd -c -b /mosquitto/config/password_file telsonbase "$MQPASS"
chmod 644 monitoring/mosquitto/password_file
docker compose restart mosquitto
```

## Docker Secrets vs `.env`

| | Docker Secrets (`./secrets/`) | `.env` File |
|---|---|---|
| **Visibility** | Not in `docker inspect` | Visible in `docker inspect` |
| **Storage in container** | tmpfs mount at `/run/secrets/` | Process environment |
| **Use for** | Keys, passwords, signing secrets | Domains, log levels, feature flags |
| **Commit to git?** | Never | Only `.env.example` (no real values) |
| **After rotation?** | Stack restart required | Synced automatically by script |

The rule is simple: if it is a secret, it goes in `./secrets/`. If it is configuration, it
goes in `.env`. The inventory table above lists everything that goes in `./secrets/`.

## Production Mode (`TELSONBASE_ENV=production`)

Set this in your `.env` before going to production:

```env
TELSONBASE_ENV=production
```

In production mode:
- **JWT secret using an insecure default** → `ValueError` raised, application refuses to start
- **Infrastructure passwords using dev defaults** (`telsonbase_dev`, `telsonbase_redis_dev`) → `warnings.warn()` at startup
- `validate_production_secrets()` in `core/config.py` runs additional checks and returns a list of any failures

The difference matters: the JWT check is a hard block. The infrastructure password checks
are warnings. Both are caught by `generate_secrets.sh` - if you ran the script, all values
are cryptographically generated and neither check will fire.

## What to Back Up

```
./secrets/          ← back up to secure off-machine storage
./backups/          ← application data backups
acme.json           ← Let's Encrypt certificates (in the letsencrypt_data Docker volume)
```

If `./secrets/` is lost and you do not have a backup, all encrypted data is unrecoverable and
all sessions are permanently invalidated. Back it up before rotation and before any
infrastructure change.

---

## What to Never Commit

```
./secrets/          ← in .gitignore
.env                ← in .gitignore (only .env.example is tracked)
acme.json           ← contains private TLS keys
monitoring/mosquitto/password_file  ← excluded from tarball, not in git
```

Verify before any git operation:

```bash
bash scripts/generate_secrets.sh --check
git status
```

If `secrets/` or `.env` appear in `git status`, stop and check `.gitignore`.

## Rotation Reminder

```env
SECRETS_ROTATION_REMINDER_DAYS=90
```

Set in `.env`. Controls how often the platform surfaces a rotation reminder in the admin
console. Default is 90 days. Adjust to match your compliance requirements - SOC 2 and HIPAA
both recommend regular key rotation; the specific interval depends on your risk assessment.
